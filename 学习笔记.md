#### *RAG*

1. *检索（Retrieval）*

   - *高效搜索和相似性检索：最大内积搜索及其变体，例如TPU-KNN、Scalab了Nearest Neighbors(ScaNN)等，通过最近邻搜索提高检索速度*
   - *模态中心检索：根据模态特性优化检索效率，包括文本中心检索（BM25、MiniLM）、视觉中心检索（EchoSight、ImgRet）和视频中心检索（iRAG、VideoRAG）*
   - *重排序和选择策略：通过优化示例选择、改进相关性评分和应用过滤机制来提高检索质量。例如：BERTScore、SSIM*

2. *融合（Fusion）*

   *融合机制的目标是将来自不同模态的数据整合到统一的表示中，以支持跨模态推理*

   - *分数融合与对齐：通过将不同模态的数据统一转换为统一格式，或嵌入到共享语义空间中，实现模态间的对齐。例如：CLIP Score或BLIP特征融合*
   - *基于注意力机制：动态甲醛跨模态交互。例如：双流共注意力机制（RAMM）和基于用户注意力的特征融合（RAGTrans）*
   - *统一框架和投影：将多模态输入整合为连贯的表示，例如IRAMIG、SAM-RAG*

3. *增强技术（Augmentation）*

   *增强技术通过优化检索到的文档，提升多模态RAG系统的性能*

   - *上下文丰富化：通过添加额外的上下文元素（如文本片段、图像标记或结构化数据）来增强检索到的知识，使其更适合生成任务*
   - *自适应和迭代搜索：根据查询的复杂性动态调整检索过程，例如多轮检索策略逐步细化检索结果（OMG-QA），通过反馈机制优化检索内容（IRAMIG）*

4. *生成方法*

   *生成方法关注与提升多模态RAG系统的输出质量和连贯性*

   - *上下文学习：利用检索到的内容作为上样本示例，增强模型的推理能力*
   - *推理：通过分解复杂推理任务为多个小步骤，提升生成内容的逻辑性和准确性*
   - *指令调整：针对特定任务调整生成模块，提升模型的指令的理解的执行能力*
   - *来源归因：确保生成内容能够追溯到具体来源，提升系统的透明度和可信度*

claude code 工作流

1. 输入入口 --> 产品外壳与交互：CLI接收用户指令，解析参数，启动主循环
2. 安全检查 --> 控制面安全：权限校验、Feature Gate检查、危险操作拦截
3. 上下文拼装-->上下文系统：System Prompt注入、历史会话加载、微压缩合并冗余消息 -->上下文折叠（可选）--> 自动压缩（超限时生成摘要）
4. 模型调用 --> 模型与云服务：调用Claude API，流式返回响应（文本或tool_use）
5. 工具调度与执行 --> 工具与扩展系统：匹配工具实现(内置30+工具执行)-->MCP外部服务接入-->结果写回消息称数组，工具调用时只读工具并行，写操作穿行
6. 循环与停止 --> Agent执行内核：检查停止条件（轮次上限、Token预算、用户推出）--> 不满足则回到3继续循环

  - Doing tasks/任务执行：不过度设计，先读后改

  `defer_loading`：API服务器端在计算Prompt cache时，会忽略带有`defer_loading`标签的工具，这意味这`defer_loading`标签的工具增减不会影响System Prompt的缓存命中

  ```json
      "type": "object",
      "properties": {

  ```

上下文缓存

如果连续请求的prompt前缀完全相同，服务器端可以缓存前缀的KV向量，下次请求时直接服用，跳过前缀处理。从前缀未命中部分开始重新计算KV向量

实现机制：

在客户端防止cache_control断点，保持prompt前缀稳定，避免不必要的变更

在模型端接收带有cache_control的请求，在标记位置建立KV cache，若命中则复用缓存

claude code如何保证prompt前缀稳定

1. System Prompt：分级缓存，分为global、org、null。

   静态区prompt为global，全球共享，在对话时，可能命中的KV cache为其他用户的静态区prompt

   使用MCP后为org，使用同一MCP Server的用户为一个org，同一个org缓存同一个KV cache

   **博客矛盾，静态区中不包含MCP instruction，MCP instrucion在动态区中，System Prompt global缓存仅缓存静态区_~Rd~_**

2. Message缓存：每条消息上标记一个ephemeral断点，当新消息追加时，之前的消息可以从缓存中读取。

四大机制提供缓存稳定性保障：

1. Section Memorization：System prompt中的动态区计算一次缓存到`/clear`、`/compact`中
2. Tool Schema Cache：会话级别缓存工具的JSON Schema序列化结果
3. Beta Header锁存：API请求头首次发送后锁定
4. MCP Instructions Delta：以Attatchment的方式，在断开或链接时更新

cache_edits（缓存生效时）：工具的输出结果超过keeprecent阈值后，不改变prompt，直接编辑服务器端中的KV cache，删除KV cache中的工具输出结果。

若距离上次调用时间长，模型端缓存失效，则直接修改本地的prompt

如何检测缓存断裂：

1. 预调用时进行状态快照，记录可能影响缓存的前缀维度，如果状态变了，则记录到pendingChanges
2. 调用后token分析。检查命中的token数量是否骤降（>5%且>2000 tokens）,此时判断cache 真断裂，根据与调用时的pendingChanges来归因。若预调用时记录的维度均为变化，但缓存还是断裂，则根据时间归因于TTL过期（>1h）或服务器路由变化(<5min)

##### 权限与沙盒

如何保证claude code 不执行类似于`rm -rf *`的破坏性命令？claude code的安全体系分为三层防线

- 权限系统

  决定”能不能执行“，选择auto模式后，仍有一个YOLO（You Only Live Once）分类器，使用一个轻量级模型判断命令是否为`atuo-allow`、`auto-deny`

  存在`denialTracking`机制，若一个工具被拒绝多次后，降级为ask，让用户做选择，防止死循环

  Bashtool的AST(Abstract Syyntax Tree)检查：对于一条Bash命令，通过AST解析成多条子命令，对多条子命令逐一进行权限匹配，经过一票否决与就高原则，同一决定是否执行该命令

  权限拒绝后的建议系统：当工具被权限系统拒绝后，系统生成一段建议，建议内容包括：命令为什么被拒绝、如果想执行可以怎么配置、配置的具体命令

- 沙盒系统

  执行隔离

- 错误处理

  出问题怎么处理

  - ShellError：Bash命令执行失败，把错误信息返回给模型，让模型根据错误信息决定下一步
  - McpAuthError：将MCP客户端的状态更新为`need-auth`，用户界面提示重新授权。
  - AbortError：被取消的错误，收到该错误后，所有并行的相关任务一并取消
  - 通用Error：发遥测（tengu_tool_use_error）、写日志（logForDebugging）、出发PostToolUseFailure Hook，把错误信息返回给模型

  ##### 主循环与状态机

  调用链：用户输入-->ask()/submitMessage()-->QueryEngine()-->query()

  QueryEngine：管理Session、构建QueryParams、调用query、把yield事件转发给消费者。

  query的输出通过两种方式消费，REPL消费（渲染到终端）、SDK消费（推流到调用方）

  `While(true)+State+continue`组成一个有限状态机。state更新时重新构建state对象，保证state转移时的原子性（要么全部更新，要么全部不更新）。State让循环变为有记忆的状态机，每轮循环读取State做决策，执行完构建新的State交给下一轮，能让Agent记住上一轮发生了什么。

  State内容：

  - 跨轮次累计内容：message、toolUseContext
  - 异常恢复追踪内容：压缩计数、OTK恢复计数、Reactive标志
  - 本轮控制：OTK 覆盖、工具摘要、Stop Hook、turnCount、transition

  主循环工作流：

  1. 准备：处理历史消息，拼装prompt
     - Snip：保留最近的消息直到累计 token 接近某个上限，更早的消息直接丢弃，计数缩减的token数，便于AotuCompact计算
     - Microcompact：同一tool_use_id的重复结果合并保留最新
     - ContextCollapse：大段完整对话折叠为摘要，可逆，可通过`projectView()`读取摘要对应的完整内容
     - AutoCompact（使用大模型）：总token超过模型上下文，调用compact agent生成摘要。执行时机：当前消息的token估算数量超过了阈值。压缩失败重试3次后放弃。执行后prompt变为摘要消息+少量保留消息（最近几个turn的对话）
     - Reactive compact：触发时机：大模型返回413：prompt_too_long，仅尝试一次。执行操作：Collapse Drain（ContextCollapse标记了可折叠的消息，此时将这些消息使用摘要替换原文）、Reactive Compact（生成摘要）
  2. 调用：调用模型，同时边调用边处理，模型还在输出，流式处理器将输出结果推流给调用方执行
  3. 执行：执行工具
  4. 转移：构建新的State对象，准备进入下一轮

  `turnCount`记录对话轮数，用作硬上限保护，防止Agent陷入无限循环、压缩周期追踪、递归深度追踪

  PTL(prompt too long)、OTK(output token Limit)

  模型异常时切换：清理废旧消息（例如思考过程）--> 清理旧工具结果 --> 重试调用 -->失败后切换模型

  OTK：升级输出上限或注入恢复消息，多轮恢复，结果拼接

#### 软件设计模式

1. 单例模式

   单例模式是一种创建型设计模式，限制一个类只能由一个实例，并且提供一个全局访问点。

   优点：资源与状态的全局唯一性、减少系统资源消耗

   缺点：违反高内聚低耦合的设计原则、单例对象的生命周期与应用相同，在单元测试中进行隔离测试变得困难

   应用场景：配置管理器、日志记录器、硬件管理器、应用状态管理

   实现方法：

   1. 模块级单例

      ```python
      # singleton_module.py
      class _Singleton:
      def __init__(self):
      self.value = 0
      def increment(self):
      self.value += 1
      return self.value
      
      singleton = _Singleton()
      
      
      # main.py
      from singleton_moudle import singleton
      
      print(singleton.increment())
      ```

      Python中的每个文件都是一个模块，模块对象在`sys.moudles`中缓存，第二次导入时，直接返回模块中的对象

      优点：线程安全

      缺点：实例在模块加载时创建，无法延迟初始化，且无法继承或扩展

      在语法上可以在`main.py`中`import _Singleton`，并创建实例与扩展，此时单例属性被破坏，模块缓存的单例实现方法只能对外部暴露`singleton`实例

   2. 线程安全的重写`__new_`方法:

      ```python
      import threading
      
      class Singleton:
          _lock = threading.Lock()
      
          def __new__(cls, *agrs, **kwargs):
              if "_instance" not in cls.__dict__:
                  with cls._lock:
                      if "_instance" not in cls.__dict__:
                          cls._instance = super().__new__(cls)
              return cls._instance
          
          def __init__(self, value = 0):
              if not hasattr(self, "_initialized"):
                  self.value = value
                  self._initialized = True
          
          def increment(self):
              self.value += 1
              return self.value
      
      ```

   3. 装饰器方法：

      ```python
      import functools
      import threading
      
      def singleton(cls):
          instances = {}
          lock = threading.Lock()
          
          @functools.wraps(cls)
          def get_instance(*args, **kwargs):
              if cls not in instances:
                  with lock:
                      if cls not in instances:
                          instances[cls] = cls(*args, **kwargs)
              return instances[cls]
          return get_instance
      
      @singleton
      class Myclass:
          def __init__(self, value=0):
              self.value = value
          def increment(self):
              self.value += 1
              return self.value
      ```

      优点：无需修改类定义

      缺点：装饰器中的闭包变量`instances`、`lock`外部无法访问，无法继承，此时`Myclass`是一个方法

   4. 元类方法：

      ```python
      import threading
      
      class SingletonMeta(type):
          _instances = {}
          _lock = threading.Lock()
      
          def __call__(cls, *args, **kwargs):
              if cls not in cls._instances:
                  with cls._lock:
                      if cls not in cls._instances: 
                          cls._instances[cls] = super().__call__(*args, **kwargs)  
              return cls._instances[cls]
      
      class Myclass(metaclass = SingletonMeta):
          def __init__(self, value = 0):
              self.value = value
          def increment(self):
              self.value += 1
              return self.value
      ```

      有点：最优方案，类无需修改

2. 工厂模式

   工厂模式是一种创建型设计模式，用于将对象的创建过程封装起来，由子类决定实例化哪一个类。

   特点：

   1. 封装性：将对象的创建过程封装在工厂类中
   2. 扩展性：通过继承和多态，可以轻松添加新的产品类
   3. 抽象性：工厂方法定义了创建对象的接口，但具体对象的创建由子类实现

   优点：

   1. 将对象的创建和使用分离，提高了模块间的独立性
   2. 易于扩展，增加新的产品类时，不需要修改现有代码，符合开闭原则

   缺点：

   1. 每增加一个产品类，就需要增加一个具体的工厂类，这可能会导致类的数量急剧增加
   2. 工厂类集中了所有实例的创建逻辑，可能导致工厂类国语庞大

   应用场景：

   1. 数据库连接：根据不同数据库类型，如`MySQL`、`PostgreSQL`，创建相应的数据库连接对象
   2. 支付网关：根据不同的支付方式，如信用卡、Paypal、微信支付，创建相应的支付处理对象
   3. 图像处理：在图像处理软件中，根据不同的文件格式，如JPEG、PNG，创建相应的图像处理器。

3. 观察者模式

   观察者模式是一种行为设计模式，他定义了对象间的一种一对多的依赖关系，使得当一个对象改变状态时，所有依赖该对象的对象都会得到通知并自动更新

   特点：

   1. 一对多关系：一个主题可以有多个观察者
   2. 抽象耦合：观察者和主题之间是抽象耦合的，增加新的观察者不会影响现有的主题
   3. 动态联动：观察者可以在任何时候加入或退出

   优点：

   1. 降低了对象之间的耦合度，主题与观察者之间是松散耦合的
   2. 扩展性好，增加新的观察者或主题类不影响现有的类

   缺点：

   1. 当观察者很多时，通知的分发可能会造成性能问题
   2. 如果观察者和主题之间的依赖关系过于复杂，会导致系统难以维护

   应用场景：

   1. 事件监听系统：在GUI应用系统中，用户组件（如按钮、文本框等）可以作为观察者，监听用户的输入
   2. 消息系统：在及时通讯软件中，当有新消息到达时，所有在线的用户（观察者）都会收到通知
   3. 资源监控：在系统监控工具中，当系统资源（如CPU、内存使用率）超过限定阈值时，监控系统会收到通知并采取相应措施

4. 装饰者模式

   装饰者模式是一种结构形设计模式，允许用户在不修改自身对象的基础上，通过添加装饰者对象动态地给对象添加额外的职责和功能

   特点：

   1. 动态扩展：可以在运行时动态地给对象添加职责或功能
   2. 透明性：装饰者模式不改变对象的接口，因此对客户端来说是透明的
   3. 灵活性：可以多个装饰者组合使用，为对象添加多个职责

   优点：

   1. 增加对象的职责是动态的、可撤销的
   2. 可以用多个装饰者包装一个对象，添加多个职责
   3. 装饰者和对象可以独立变化，不会相互耦合

   缺点：

   1. 过度使用装饰者模式可能会使系统变得复杂，难以理解

   2. 可能引起多层装饰者调用，影响性能

   应用场景：

   1. 日志记录：在不修改原有对象的基础上，添加日志记录功能
   2. 安全控制：为对象添加访问控制
   3. 资源管理：为对象添加额外的资源管理功能

   装饰者模式与Python语法糖装饰器的辨析

   | 作用对象 | 对象(Object)，对对象的实例进行包装             | 函数或类，在定义时加`@`                                  |
   | 实现机制 | 组合。包装器持有被包装者的引用，调用时层层传递 | 高阶函数。装饰器接收一个函数或类作为参数，返回一个新函数 |
   | 执行时机 | 运行时                                         | 定义时                                                   |

5. 策略模式

   策略是一种行为设计模式，它定义了一系列算法，并将每一个算法封装起来，使它们可以互相替换。策略模式让算法独立于使用它的客户端

   特点：

   1. 封装变化：将变化的部分封装起来，与稳定部分分离
   2. 多态性：使用多态来实现算法的替换
   3. 替换性：在运行时选择使用哪个策略

   优点：

   1. 算法的变化独立于使用算法的客户端
   2. 容易添加新算法而不影响客户端

   缺点：

   1. 客户端必须了解所有策略类的差异，以便使用适当的策略

   应用场景：

   1. 算法选择：在应用程序中，根据不同的业务需求选择不同的算法
   2. 排序算法：根据不同的数据特性选择不同的算法
   3. 路径查找：在地图服务中，根据不同的优化标准（时间最短、距离最短）选择不同的路径查找算法

   >###### 代码实现
   >
   >```python

6. 适配器模式

   适配器模式使一种结构形设计模式，用于使原本不兼容的接口能够一起工作。它通常设计到一个客户端使用一个期望的特定接口，而另一个类或组件提供了一个不同的接口。适配器模式通过创建一个中间层（适配器），将一个类的接口转换成客户端期望的另一个接口

   特点：

   1. 接口转换：适配器模式提供了将一个类的接口转换成另一种接口的方式
   2. 兼容性：解决了接口不兼容的问题，使得原本不能在一起工作的类可以协同工作

   优点：

   1. 增加了类的兼容性，使得它们可以一起工作，即使它们接口不兼容
   2. 客户端代码不需要修改，通过适配器与目标交互

   缺点：

   1. 过度使用适配器模式可能会使系统变得复杂，难以理解和维护
   2. 适配器可能会引入性能开销，尤其是需要频繁调用适配器方法的情况下

   应用场景：不同系统的继承、第三方库的继承、硬件设备控制

   ```python
   from abc import ABC, abstractmethod
   
   class NewPayment(ABC):
       @abstractmethod
       def charge(self, money):
           pass
   
   class OldPayment:
       def __init__(self):
           super().__init__()
   
       def pay(self, num, currency):
           return f"Pay {num}{currency}"
       
   class NewSystem:
       def __init__(self, payment: NewPayment):
           super().__init__()
           self.payment = payment
   
       def charge(self, money):
           return self.payment.charge(money)
   
   class PayAdapter(NewPayment):
       def __init__(self, old_payment: OldPayment):
           super().__init__()
           self.old_payment = old_payment
       def charge(self, money):
           return self.old_payment.pay(money, '$')
       
   if __name__ == "__main__":
       old_payment = OldPayment()
       payment_adapter = PayAdapter(old_payment)
       system = NewSystem(payment_adapter)
       print(system.charge(10))
   ```

7. 代理模式

   代理模式使一种结构形设计模式，它为另一个对象了提供了一个替代或占位符，以控制对它的访问。代理可以在不改变对象的代码的前提下，通过引入代理对象来间接访问原始对象，从而在不直接暴露原始对象的情况下，提供额外的功能操作。

   特点：

   1. 间接访问：通过代理对象来间接访问原始对象
   2. 职责分离：将控制逻辑与业务逻辑分离，代理对象负责控制逻辑，原始对象负责业务逻辑
   3. 延迟初始化：代理可以在需要时才创建对象，实现延迟初始化

   优点：

   1. 降低了系统的耦合度，增强了对象的可控性
   2. 可以为原始对象提供额外的安全控制或延迟加载等操作
   3. 增加了代码的可扩展性，可以在不修改原始对象的情况下，通过引入新的代理类扩展功能

   缺点：

   1. 增加了系统的复杂性，可能让系统设计变得更加复杂
   2. 可能会引入性能开销，尤其是在代理对象需要进行复杂控制逻辑时

   应用场景：访问控制

   ```python
   import time
   
   class HighresolutionImage:
       def __init__(self, image_name):
           self.image_name = image_name
           self.load_form_disk()
   
       def load_form_disk(self):
           print(f"模拟从磁盘读取高清图片，耗时2s")
           time.sleep(2)
       
       def display(self):
           print("display Highresolution Image")
   
       def rotate(self, degree):
           print(f"Rotate image {degree}")
       
   class ImageProxy:
       def __init__(self, file_name):
           self.image_name = file_name
           self.real_image = None
       
       def display(self):
           if self.real_image is None:
               self.real_image = HighresolutionImage(self.image_name)
           self.real_image.display()
       
       def __getattr__(self, name):
           if self.real_image is None:
               self.real_image = HighresolutionImage(self.image_name)
           return getattr(self.real_image, name)
   
   if __name__ == "__main__":
       image_proxy = ImageProxy("Rose")
       image_proxy.rotate(degree = 80)
       image_proxy.display()
   ```

   Pythonic的写法相比于OOP写法，不需要继承，因为Python中存在`__getattr__`魔法方法，使得不继承也可以寻找到组合对象的属性和方法

9. 组合模式

   组合模式是一种结构形设计模式，它允许你将对象组合成树状结构，以表示“部分-整体”的层次结构。这种模式使得用户可以一致地对待单个对象和对象组合

   特点：

   1. 部分-整体层次结构：可以包含其他组合或叶节点，形成树状结构
   2. 一致性：客户端代码可以一致地处理组合结构和叶节点

   优点：

   1. 简化了客户端代码，客户端可以同一处理组合结构和对象
   2. 更好的层次结构表示，易于扩展和维护

   缺点：设计比较复杂，需要合理地设计组件的接口和类

   应用场景：

   1. 文件系统：文件系统中的文件和文件夹可以形成树状结构，其中文件夹可以包含其他文件和其他文件夹
   2. 组织结构：公司的组织结构可以表示成树状结构，其中每个部门包含员工和其他子部门
   3. 分布式系统：在分布式系统中，资源可以组合成树状结构，以方便管理和访问

   ```python
   from abc import ABC, abstractmethod
   
   class Employee(ABC):
       @abstractmethod
       def get_salary(self):
           pass
   
   class Worker(Employee):
       def __init__(self, name: str, salary: float):
           super().__init__()
           self.name = name
           self.salary = salary
   
       def get_salary(self) -> float:
           return self.salary
       
       def __add__(self, other) -> float:
           if isinstance(other, Employee):
               return self.salary + other.get_salary()
           return self.get_salary() + other
       
       def __radd__(self, other) -> float:
           return self.__add__(other)
       
   class Department(Employee):
       def __init__(self, name):
           super().__init__()
           self.name = name
           self._children = []
   
       def add(self, employee:Employee):
           self._children.append(employee)
       
       def get_salary(self):
           return sum(self._children)
       
       def __add__(self, other):
           if isinstance(other, Employee):
               return self.get_salary() + other.get_salary()
           return self.get_salary() + other
       
       def __radd__(self, other):
           return self.__add__(other)
       
       def __iter__(self):
           return iter(self._children)
       
   if __name__ == "__main__":
       worker_a = Worker("a", 3.5)
       worker_b = Worker("b", 1.7)
       worker_c = Worker("c", 2.8)
   
       tech = Department("tech")
       tech.add(worker_a)
       tech.add(worker_b)
   
       company = Department("company")
       company.add(worker_a)
       company.add(tech)
       print(company.get_salary())
       print(company._children)
   ```

10. 迭代器模式

    迭代器模式是一种行为设计模式，它允许你顺序访问一个聚会对象中的各个元素而不需要暴露其内部的表示。迭代器模式提供了一种通过抽象迭代器来遍历元素的方法，使得你可以在不知道具体集合类型的情况下，对集合进行遍历。

    特点：

    1. 抽象化遍历过程：迭代器定义了遍历元素的接口
    2. 支持多种遍历方式：不同的迭代器可以实现不同的遍历策略
    3. 聚合对象与迭代器解耦：聚合对象不需要知道迭代器的具体实现

    优点：

    1. 抽象化集合的访问，使客户端代码与集合内部的表示无关
    2. 可以提供多种遍历方式，如正序或逆序遍历
    3. 增加了集合的灵活性，可以在不修改集合类的情况下，引入新的遍历方式

    缺点：

    1. 增加了系统的复杂性，需要为每个聚合类设计迭代器
    2. 需要额外的代码来实现迭代器

    应用场景：遍历集合、数据结构、数据库查询

    ```python
    class Book:
        def __init__(self, title):
            self.title = title
        
        def __repr__(self):
            return f"Book{self.title}"
    
    class Bookshelf:
        def __init__(self):
            self._books = []
    
        def add_book(self, book: Book):
            self._books.append(book)
    
        def __iter__(self):
            for book in self._books:
                yield book
    
        def reversed_iter(self):
            for book in reversed(self._books):
                yield book
    
    if __name__ == "__main__":
        shelf = Bookshelf()
        shelf.add_book(Book("三体"))
        shelf.add_book(Book("流浪地球"))
        shelf.add_book(Book("乡村教师"))
    
        print("=== 正序（生成器） ===")
        for book in shelf:
            print(book)
    
        print("\n=== 倒序（生成器） ===")
        for book in shelf.reversed_iter():
            print(book)
    ```

2. python的`cls.__dict__`

   `cls.__dict__`是一个只读映射，存储当前类自身定义的所有属性和方法，不包含从父类继承的东西

   `hasattr`与`cls.xxx`查找都会沿着继承链查找，`cls.__dict__`仅看当前类，不会查找继承链

   作用：可区分类属性与实例属性，调用`实例.__dict__`查看实例的属性，调用`类.__dict__`查看类属性，两者互不干扰

   特点：

   1. 只读，不可通过字典`[]`的方式修改，但可通过`.`的访问修改，例如`cls.__dict__['_instance'] = None`不成功，但`cls._instance = None`可成功
   2. 不包含`@property`或`@classmethod`装饰器算出的动态属性，仅包含对象本身。**待深入~rd~**

6. 闭包与装饰器

   闭包是指一个函数捕获了其外部作用域中的自由变量，即使外层函数执行完毕，这个内层函数依然持有对这些变量的引用。

   延迟绑定特性：闭包引用的外部变量时按引用捕获的。也就是说，闭包内部存储的变量时内存地址，而不是变量当时的值，当闭包被执行时，他才会去这个地址读取变量的当前值。

   由于延迟绑定特性，在循环中使用闭包时，执行时刻的变量值将会覆盖原本预期的变量值，可通过默认参数的方式避免覆盖。

   为什么默认参数可以避免覆盖？因为python传参时均传递对象地址。默认参数在函数定义时计算其参数值，具体过程为，在函数对应的栈上创建形参指针，让指针指向传入的变量的地址。当外部变量被循环覆盖时，上一次循环的闭包函数的形参指针已经指向上一轮循环的变量地址，本次循环不会覆盖上一轮循环的闭包函数的形参指针。

   相关知识点：默认参数仅在定义函数时计算一次，考虑以下场景：

   ```python
   def func(l=[]):
       l.append(1)
       return l
   
   if __name__ == "__main__":
       primt(func())
       print(func())
       
   # 预期输出：
   # l:[1]
   # l:[1,1]
   ```

   由于默认参数仅计算一次，所以不同次调用的`func`函数均对同一个列表进行操作

   类中的方法受到`@abstractmethod`装饰时，继承该类的子类必须实现抽象方法，否则实例化时报错

   类中的方法受到`@staticmethod`装饰时，表示该方法为静态方法，属于类的方法，而非实例的方法

7. 抽象类

   继承`ABC`的类为抽象类，当抽象类中存在`@abstractmethod`时，该类不能实例化；反之，抽象类中不包含`@abstractmethod`时，可以实例化。 该类的子类没有重写所有`@abstractmethod`时，不可实例化

   可以通过`cls.__abstractmethods__`查看当前类还有哪些抽象方法未实现

8. 魔法方法

   1. `__init_subclass__`在子类对象被创建时执行。子类的创建在执行到代码`class Myclass`时执行，子类的实例化在`myclass = Myclass()`时执行，是不同的两个概念
   2. `__getattr__`通过实例`self.xxx`访问属性时，遵循优先级：类及其父类中的数据描述符-->``self.__dict__`-->类及其父类中的非数据描述符-->`self.__getattr__`方法
   3. `__add__`与`__radd__`：对象实现了`__add__`（左加）与`__radd__`（右加）后，可以使用`+`运算符将两个对象相加，执行`a + b`时，调用顺序为`a.__add__(b)`，若失败，则调用`b.__radd__(a)`

9. `super().__init__()`

   子类的`__init__()`方法中调用`super().__init__()`是继承父类在`__init__()`中定义的实例属性的唯一手段，若子类不调用`super().__init__()`则丢失这部分属性，这部分属性存储在`self.__dict__`中

10. `yield`

    函数中包含`yield`时，该函数的类型仍然为`function`，但调用该函数时，返回一个`generator`，继承`generator`中的一些方法，例如`send()`、`throw`、`close()`等。调用函数时，不执行函数的函数体，返回一个`generator`对象，当调用到`next(gen)`时(在`for x in gen`时自动调用`next`)，进入函数，开始执行函数体，执行到`yield`时暂停，下一次调用`next`时，从`yield`恢复执行。

    `yield`、`finally`：当`yield`在`try...finally`中时，`finaly`代码的执行时机：

    1. 正常耗尽，也即是迭代完成
    2. 显示关闭，调用`gen.close()`时，执行`finally`块代码
    3. 垃圾回收，在生成器对象没有任何引用时，执行`finally`块代码。需要注意，在`for`循环中`break`后，若生成器对象仍存在引用，则不会执行`finally`模块

    若`yield`右侧是一个`generator`对象，则可以使用`yield from`完成生成器的嵌套

    `send()`、`throw()`：语句`val = yield total`在编译时看作两条指令，在产出时，执行`yield total`，产出`total`，在`send()`接收时执行`val = yield`，将收到的值传递给`val`，然后程序继续执行
